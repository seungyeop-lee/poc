# 현재 구현 분석 및 요구사항 정의

## 현재 구현 분석

### 문제점
현재 PreviewCanvasComponent (frontend/src/PreviewCanvasComponent.tsx)에서는:
- **이미지 전체 영역이 드래그 가능 영역**: 87-109라인의 `<div>` 컨테이너가 마우스 이벤트를 처리
- **이미지와 드래그 영역이 겹침**: 이미지를 보면서 드래그하기 불편
- **시각적 피드백 부족**: 어디를 드래그해야 하는지 명확하지 않음

### 현재 드래그 메커니즘
1. `handleMouseDown` (60-63라인): 드래그 시작 지점 기록
2. `handleMouseMove` (65-76라인): 마우스 이동에 따라 `settings.left`, `settings.top` 업데이트
3. `handleMouseUp` (78-80라인): 드래그 종료
4. 컨테이너 div에 `onMouseDown`, `onMouseMove`, `onMouseUp` 이벤트 핸들러 바인딩

## 요구사항 정의

### 개선 방향
별도의 드래그 전용 영역을 만들어 이미지와 분리:
1. **이미지는 읽기 전용**: 이미지 자체는 드래그 불가능하게 변경
2. **별도 드래그 핸들 영역**: 작고 명확한 드래그 핸들을 추가
3. **시각적 구분**: 드래그 가능한 영역을 시각적으로 명확하게 표시

### 제안 UI 패턴
**옵션 1: 코너 드래그 핸들**
- 이미지 위에 작은 드래그 핸들 표시 (예: 우상단 코너)
- 핸들만 드래그 가능, 이미지는 클릭 불가

**옵션 2: 별도 드래그 영역**
- 이미지 하단에 드래그 전용 영역 추가
- "Drag here to adjust text position" 안내 문구

**옵션 3: 텍스트 위치에 드래그 핸들**
- 텍스트가 표시된 위치에 투명한 드래그 핸들 오버레이
- 텍스트를 직접 드래그하는 느낌

### 선택된 패턴
**옵션 1 (코너 드래그 핸들)** 선택:
- 이유: 가장 직관적이고 이미지 영역을 최소한으로 가림
- 구현 방법:
  - 이미지 위에 absolute position으로 작은 드래그 핸들 배치
  - 핸들 위치는 `settings.left`, `settings.top`에 따라 동적으로 계산
  - 핸들만 드래그 이벤트 처리, 이미지 컨테이너는 드래그 불가

## 구현 계획
1. 이미지 컨테이너에서 드래그 이벤트 핸들러 제거
2. 텍스트 위치에 드래그 핸들 추가 (absolute positioning)
3. 드래그 핸들에만 드래그 이벤트 핸들러 바인딩
4. 핸들 스타일링: 명확하고 눈에 잘 띄게 (예: 작은 사각형, 십자가 아이콘 등)
